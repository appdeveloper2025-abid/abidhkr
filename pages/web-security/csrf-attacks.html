Cross-Site Request Forgery (CSRF)

Introduction to CSRF

Cross-Site Request Forgery (CSRF) is an attack that forces an end user to execute unwanted actions on a web application in which they're currently authenticated.

Core Theory

CSRF exploits the trust that a website has in a user's browser. Think of it like someone forging your signature on a check - the bank trusts the signature because it looks legitimate, even though you didn't actually sign it. Similarly, websites trust requests from your browser because they include your authentication cookies.

How CSRF Works - Technical Deep Dive

CSRF attacks exploit the automatic inclusion of credentials in HTTP requests:

  1. User Authentication: Victim logs into a legitimate website (bank.com)
  2. Session Establishment: Browser stores authentication cookies
  3. Malicious Site Visit: User visits attacker's website while still logged in
  4. Forged Request: Attacker's site triggers request to bank.com
  5. Automatic Credentials: Browser automatically includes session cookies
  6. Unauthorized Action: Bank processes request as legitimate

Browser Security Model Context

Why Browsers Are Vulnerable
  • Automatic Cookie Inclusion: Browsers send cookies with every request to a domain
  • Same-Origin Policy Limitations: SOP doesn't prevent sending requests, only reading responses
  • Cross-Origin Requests: Forms and images can be loaded from any domain
  • HTTP Authentication: Basic/Digest auth automatically included
  • Client Certificates: Automatically presented for HTTPS sites

CSRF Attack Vectors

Traditional Vectors

  • HTML Forms: Hidden forms that auto-submit
  • Image Tags: GET requests via img src
  • JavaScript: Dynamic form creation and submission
  • IFrames: Hidden frames containing malicious forms

Modern Vectors

  • AJAX Requests: XMLHttpRequest and Fetch API
  • WebSockets: Real-time communication protocols
  • Flash/Java: Plugin-based cross-domain requests
  • Mobile Apps: WebView components in mobile applications

Impact Assessment

CSRF attacks can have severe consequences depending on the application functionality:

  • Financial Fraud: Unauthorized money transfers, purchases
  • Account Takeover: Password changes, email modifications
  • Data Manipulation: Unauthorized data modifications
  • Privilege Escalation: Administrative action execution
  • Social Engineering: Reputation damage through unwanted posts

Real-World Examples

  • Banking: Transferring funds to attacker's account
  • Social Media: Posting malicious content or sending messages
  • E-commerce: Changing shipping addresses or making purchases
  • Admin Panels: Creating new admin users or modifying settings

Key Characteristics

  • State-Changing Actions: Targets actions that modify data
  • Authentication Bypass: Uses victim's existing session
  • Social Engineering: Requires user interaction
  • Cross-Origin: Attack originates from different domain
  • Silent Execution: Victim unaware of attack
  • Browser-Based: Exploits browser behavior
  • Session Hijacking: Leverages active sessions
  • Privilege Escalation: Uses victim's permissions

How CSRF Attacks Work

Attack Flow

# CSRF Attack Steps 1. User logs into vulnerable website (bank.com) 2. User receives session cookie 3. Attacker crafts malicious page with CSRF payload 4. User visits attacker's page while still logged in 5. Browser automatically sends cookies with forged request 6. Server processes request as legitimate (has valid session) 7. Unauthorized action is performed

Vulnerable Request Example

# Legitimate money transfer request POST /transfer HTTP/1.1 Host: bank.com Cookie: sessionid=abc123 Content-Type: application/x-www-form-urlencoded to_account=12345&amount=1000&csrf_token=missing # Same request can be forged from attacker site # Browser will automatically include the session cookie

Prerequisites for CSRF

Required Conditions
  • Relevant action exists (state-changing)
  • Cookie-based session handling
  • No unpredictable request parameters
  • User is authenticated
Attack Requirements
  • Social engineering vector
  • Victim visits attacker site
  • Active session on target site
  • Knowledge of request structure

SameSite Cookies

# SameSite cookie attributes Set-Cookie: sessionid=abc123; SameSite=Strict; Secure; HttpOnly # SameSite options: # Strict - Cookie only sent for same-site requests # Lax - Cookie sent for top-level navigation (default) # None - Cookie sent for all cross-site requests (requires Secure) # PHP implementation setcookie('sessionid', $session_id, [ 'expires' => time() + 3600, 'path' => '/', 'domain' => '.example.com', 'secure' => true, 'httponly' => true, 'samesite' => 'Strict' ]); # Express.js app.use(session({ cookie: { sameSite: 'strict', secure: true, httpOnly: true } })); # Nginx configuration add_header Set-Cookie "sessionid=$cookie_sessionid; SameSite=Strict; Secure; HttpOnly";

Custom Headers

# Require custom header for AJAX requests # JavaScript client fetch('/api/transfer', { method: 'POST', headers: { 'Content-Type': 'application/json', 'X-Requested-With': 'XMLHttpRequest' }, body: JSON.stringify(data) }); # Server validation (PHP) if (!isset($_SERVER['HTTP_X_REQUESTED_WITH']) || $_SERVER['HTTP_X_REQUESTED_WITH'] !== 'XMLHttpRequest') { http_response_code(403); die('Invalid request'); } # Double Submit Cookie pattern # Set CSRF cookie setcookie('csrf_token', $csrf_token, ['samesite' => 'Strict']); # Validate cookie matches form token if ($_COOKIE['csrf_token'] !== $_POST['csrf_token']) { die('CSRF validation failed'); }

Additional Security Measures

# Referer header validation function validate_referer() { $referer = $_SERVER['HTTP_REFERER'] ?? ''; $host = $_SERVER['HTTP_HOST']; if (strpos($referer, 'https://' . $host) !== 0) { return false; } return true; } # Origin header validation function validate_origin() { $origin = $_SERVER['HTTP_ORIGIN'] ??
